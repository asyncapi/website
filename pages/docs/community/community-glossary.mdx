This glossary provides simple definitions of common terms used across the AsyncAPI community.  
It helps new contributors understand important words, roles, and concepts they may encounter while reading the documentation or participating in discussions.

> ⚠️ Note: This document is continuously improving. More terms may be added over time.

---

# Glossary

## **A**

### **API (Application Programming Interface)**
A set of rules that allows software systems to communicate with each other.

### **Application**
A program or service that produces or consumes messages in an event-driven system.

### **Application Developer**
A person who builds applications that use AsyncAPI specifications or event-driven architecture.

### **AsyncAPI Document**
A structured file (usually JSON or YAML) that describes an event-driven API, including messages, channels, servers, and protocols.

---

## **B**

### **Broker**
A system that receives, stores, routes, and delivers messages between producers and consumers.  
Examples: Kafka, RabbitMQ, MQTT broker.

---

## **C**

### **Channel**
A communication path through which messages are sent or received.

### **CLI (Command Line Interface)**
A tool used via terminal commands to generate files, validate AsyncAPI documents, or automate workflows.

### **Consumer**
An application or service that receives messages from a channel.

### **Contribution**
Any improvement to documentation, code, tools, design, or community work in the AsyncAPI ecosystem.

---

## **D**

### **Documentation Contribution**
Updating content, fixing issues, improving clarity, or adding examples to help users understand AsyncAPI better.

---

## **E**

### **Ecosystem**
All tools, libraries, guidelines, and community projects built around AsyncAPI.

### **Event-Driven Architecture**
A design pattern where systems communicate by producing and reacting to events (messages).

---

## **G**

### **Governance**
Rules and processes that guide how the AsyncAPI community makes decisions and manages projects.

---

## **L**

### **Library**
A reusable collection of code that provides specific functionality for working with AsyncAPI or message-based systems.

---

## **M**

### **Message**
A piece of data sent from a producer to a consumer through a channel.

### **Middleware**
Software that connects different systems, enabling communication, message processing, or routing.

---

## **P**

### **Parser**
A tool that reads and validates an AsyncAPI document and provides an object representation for further tooling.

### **Protocol**
A communication standard used by servers and clients to exchange messages.  
Examples: MQTT, Kafka, WebSocket.

### **Publisher**
See **Producer**.

### **Producer**
An application that sends messages to a channel.

---

## **R**

### **REST API**
A traditional request-response API model using HTTP.  
(NOT event-driven, but included for comparison.)

---

## **S**

### **Schema**
A structured definition that describes the shape and data types of a message.

### **Server**
A messaging broker or system that connects producers and consumers and manages message distribution.

### **Specification (Spec)**
The AsyncAPI standard that defines how event-driven APIs should be described.

---

## **T**

### **Template**
A generator pattern used by the AsyncAPI Generator to create documentation, code, or other output from an AsyncAPI document.

### **Topic**
A named channel used in messaging systems (commonly used in Kafka and MQTT).

---

## **V**

### **Validation**
The process of checking whether an AsyncAPI document follows the official specification.
